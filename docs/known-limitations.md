# InterMed v8.0 Alpha Snapshot Known Limitations

This document lists the known boundaries of the current `v8.0-alpha-snapshot`.
It is intentionally conservative: if an area has not been field-tested, it is
listed here even when code exists in the repository.

## Scope

- Minecraft scope is frozen at `1.20.1`.
- Loader families in alpha scope are `Fabric`, `Forge`, and `NeoForge`.
- The project does not currently claim production stability, `95%` public-mod
  compatibility, general `Minecraft 1.20+` compatibility, or field-proven
  hostile-mod security.
- The canonical runtime path is `InterMedKernel -> LifecycleManager -> LazyInterMedClassLoader`.

## Compatibility

- Full Fabric API, Forge API, and NeoForge API parity is not yet proven.
- The `compat-corpus` command records local candidate JARs plus reporting-only
  data/resource pack ZIP archives and manifest-derived metadata. It does not
  prove that those candidates boot, reload, or behave correctly.
- The `compat-sweep-matrix` command normalizes stored harness outcomes, but it
  still does not prove gameplay, multiplayer, strict-security, or field behavior.
- Diagnostics bundles include harness-linked sweep results only when a harness
  `results.json` is provided through `--harness-results` or discovered in the
  current game directory. Otherwise the sweep matrix remains `corpus-only-not-run`.
- The `launch-readiness-report` command checks evidence artifact presence and
  docs guardrails only. It does not execute Gradle gates and does not prove
  external alpha, beta, or stable readiness by itself.
- The `api-gap-matrix` command tracks a curated alpha bridge surface and known
  beta gaps. Symbols absent from that matrix are not implied to be supported.
- As of the `2026-04-20` runtime delta, the curated alpha-stage matrix is
  `87/87` present, beta-stage tracked symbols are `5/5` present, and NeoForge
  is `17/17` present. This is still a curated surface check, not full API parity.
- Harness reports currently prove startup/boot behavior under documented
  conditions; they do not prove real gameplay, multiplayer, or long-session
  correctness.
- Public-mod Mixin stacks are still below `Field-tested`. Unsupported or unusual
  Mixin features may fail safely or require a targeted bridge/fork update.
- Reflection remapping covers common string/reflection patterns, but names built
  from user input, config files, network payloads, or complex runtime dataflow may
  still need explicit InterMed remapping API calls.
- Optional mod integrations that rely on reflection are handled through weak DAG
  edges only when the target exposes a safe API surface.

## Minecraft Lifecycle

- VFS/data-pack support is alpha-level. Classpath and overlay routing have
  in-repo evidence, and data/resource pack ZIPs are now represented in
  reporting evidence, but the full Minecraft ResourceManager/DataPack reload
  lifecycle still needs real game validation.
- Registry virtualization has synthetic and hot-path evidence, but real
  conflict-heavy modpacks and multiplayer registry sync are still deferred.
- Network bridge tests cover synthetic packet registration/dispatch. Real
  client-server handshake, login, registry sync, custom payload exchange, and
  play-session behavior need external validation.

## Security

- Strict mode is the only security-relevant lane. Permissive compatibility
  sweeps must never be cited as secure-by-default proof.
- Capability checks now fail closed for unattributed non-host callers, but
  hostile public-mod validation is still required.
- Native/JNI/JNA behavior has in-repo singleton-routing and dedup/conflict
  diagnostics, but real native-heavy public mods still need field-style testing.
- Espresso and Wasm sandboxes are intended for high-level logic, not hot render,
  world-tick, or entity/block critical paths.
- Absolute guarantees against arbitrary hostile mods are not claimed.

## Performance

- Microbench and soak gates exist in-repo, but the `10-15%` overhead target is
  not claimed until native Forge/Fabric/NeoForge baseline comparisons exist.
- Startup behavior with warm AOT cache has synthetic evidence. Real modpack-scale
  cold/warm startup measurements are still required.
- Long-running 48-hour client/server pack soaks are not yet field evidence.

## Launch And Configuration

- The active launch model is JVM arguments, existing launcher profiles, and the
  thin `InterMedLauncher`. A dedicated GUI launcher is deferred.
- Active configuration uses `intermed-runtime.properties` plus JSON override
  files. TOML/YAML and GUI override flows are deferred.
- Java 21 is required. Launchers that silently select Java 8/17 are out of scope
  until configured to run Java 21.

## Reporting

Use [alpha-triage.md](alpha-triage.md) before filing issues. The most useful
artifact is a `diagnostics-bundle` zip generated by `InterMedLauncher launch` on
non-zero exit or manually with the `diagnostics-bundle` command. For API-only
triage, use the `api-gap-matrix` command and attach the generated JSON. For
corpus planning or CI reproducibility, use `compat-corpus` and
`compat-sweep-matrix`.
